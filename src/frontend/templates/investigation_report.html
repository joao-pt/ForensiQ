Crítico

FQ-01 IsAgent/IsExpert crashes com AttributeError em superutilizadores sem o campo profile Crítico

core/permissions.py:29 core/permissions.py:43 core/permissions.py:60

IsAgent.has_permission e IsAgentOrExpert.has_permission fazem request.user.profile == 'AGENT' diretamente. O modelo User tem sempre o campo (é AbstractUser + profile com default AGENT), mas createsuperuser cria o registo sem passar por formulário e, conjugado com o guarda hasattr(user, 'profile') usado noutros sítios, torna óbvio que vocês já anteciparam utilizadores sem este atributo. Um superuser cujo profile ficasse em branco (ou qualquer AnonymousUser que escape pela IsAuthenticated anterior devido a ordem de classes) provoca AttributeError → HTTP 500 em vez de 403.

Mais grave: IsAgentOrExpert não tem a guarda de safe methods — um AGENT/EXPERT sem perfil falha na leitura também, enquanto qualquer outro utilizador autenticado é bloqueado de ler. A intenção documentada ("Leitura é permitida a qualquer utilizador autenticado com perfil válido") contradiz o código.

Usar getattr(request.user, 'profile', None) em todas as três classes, e mover o SAFE_METHODS bypass para dentro de IsAgentOrExpert.

# permissions.py — IsAgentOrExpert
def has_permission(self, request, view):
    if not request.user or not request.user.is_authenticated:
        return False
    return request.user.profile in ('AGENT', 'EXPERT')
# sem bypass de SAFE_METHODS e sem getattr — quebra se profile faltar

Alto

FQ-02 Rate limit de autenticação é desligado quando TESTING é verdadeiro — e o sinal de TESTING é promíscuo Alto

forensiq_project/settings.py:17-24, 145-157

A detecção de TESTING inclui os.environ.get('DJANGO_TESTING', '').lower() == 'true'. Isto significa que qualquer contexto onde essa env var seja definida (CI, imagem partilhada, container de staging configurado por engano) desactiva throttling, HSTS e SSL redirect. O AuthRateThrottle contra força bruta também fica inócuo: os rates passam para 10000/minute e os globais para [].

Adicionalmente, pytest aparecer em sys.argv[0] funciona só para o binário; corridas via python -m pytest não batem nesse teste e caem nas verificações de ambiente — misturar detecção por argv e por env var é frágil.

Confiar apenas no test runner: usar test_settings.py como DJANGO_SETTINGS_MODULE e remover a heurística runtime. Se mantiverem a env var, renomeá-la para algo que só exista no conftest.py (ex: FORENSIQ_TESTING_RUNNER=1) e garantir que nunca é herdada por produção.

FQ-03 Endpoints de lookup externos cacheam IMEI globalmente, contornando a permissão por perfil Alto

core/views.py — EvidenceIMEILookupView.get

O cache_key é lookup:imei:<imei>. A verificação de perfil (_user_can_lookup) corre antes, portanto o acesso controlado está ok. Mas a resposta cacheada inclui _cached_at e o próprio payload externo é guardado sem considerar o utilizador que o gerou. Em si não é leak — é ok — mas dois riscos concretos:

(1) Se alguma vez lookup_imei incluir dados derivados do request (ex. IP do agente para rate limit upstream, ou idioma), ficarão gravados 30 dias globalmente. (2) O campo cached_at retornado ao cliente é o carimbo do primeiro requisitante — qualquer auditoria ISO 27037 sobre "quando foi consultado o IMEI para esta evidência" tem de ir ao AuditLog, não ao payload. Isto é subtil mas cabe documentar explicitamente.

O maior problema prático: não há invalidação manual. Se a imeidb.xyz corrigir um TAC errado, os agentes vão ver a resposta errada durante 30 dias sem forma de forçar refresh.

Adicionar um parâmetro ?refresh=1 gated por perfil EXPERT/staff que salte a cache e reescreva a entrada. Registar no AuditLog sempre que for usado.

FQ-04 CSRF_COOKIE_HTTPONLY = False é intencional, mas SameSite=Strict parte a recuperação de sessão cross-tab Alto

forensiq_project/settings.py — CSRF section

SameSite=Strict em CSRF_COOKIE_SAMESITE e em SESSION_COOKIE_SAMESITE significa que seguir um link externo (email de notificação de nova custódia, SSO de entidade pública) para o ForensiQ chega sem os cookies. O utilizador vai aterrar em /login/ mesmo estando autenticado na mesma origem — fricção séria para first responders em campo. Lax é o padrão de facto para aplicações protegidas por CSRF token em headers (é o que fazem).

Também: ter CSRF_COOKIE_HTTPONLY=False é correcto aqui (JS precisa de ler o token para o enviar no X-CSRFToken), mas o comentário "SameSite=Strict já bloqueia" só protege cross-site — não protege contra XSS same-origin, e é isso que HTTPOnly é. Se a aplicação servir conteúdo de utilizador (observations, descrições em livre-texto) e o CSP falhar num injectar inline, o atacante pode ler o CSRF token. A decisão é aceitável mas o comentário é enganador.

Relaxar para SameSite=Lax em ambos os cookies (o JWT é só lido pelo backend; o CSRF continua protegido pela double-submit). Actualizar o comentário em settings.py.

FQ-05 A variação de fallback de _lookup_previous_hash fora do select_for_update reintroduz a race condition que tentaram corrigir Alto

core/models.py — ChainOfCustody.compute_record_hash / _lookup_previous_hash

O docstring diz explicitamente que o fallback é "apenas para compatibilidade com utilizadores antigos" e expõe a race. Mas a função continua pública no modelo e reutilizável. Qualquer chamador futuro (comandos de management, tarefas assíncronas, scripts de re-hash) que chame compute_record_hash() sem argumento vai silenciosamente desactivar a protecção, sem aviso. Um assert ou warning não chega — a forma correcta é tornar o parâmetro obrigatório e ter um método diferente (compute_record_hash_unsafe_legacy) para o fallback, para que o nome grite no code review.

Tornar previous_hash um argumento obrigatório. Mover o fallback para um método nomeado à parte. Adicionar warnings.warn(DeprecationWarning) se for chamado.

Médio

FQ-06 Evidence.save() chama full_clean() antes de calcular o hash — mas integrity_hash não aparece nos read_only_fields do serializer Médio

core/serializers.py — EvidenceSerializer.Meta.read_only_fields

integrity_hash está listado como read-only — correcto. Mas o serializer também marca timestamp_seizure como read-only, o que entra em contradição com o evidences_new.html típico (o agente indica quando apreendeu a evidência, não é o momento em que clica "submeter"). Se o fluxo UX expuser um campo de data/hora de apreensão, o valor é silenciosamente descartado na API.

Clarificar: ou o campo é read-only (e o form frontend não o mostra) ou é writable com validação "não pode ser futuro" (que já existe no clean). Confirmar com o form.

FQ-07 DashboardStatsView: list(closed_occurrence_ids) materializa em memória sem limite Médio

core/views.py — DashboardStatsView.get

Envolver uma values_list distinta num list(...) para usar em exclude(id__in=...) funciona para uma demo, mas com N ocorrências com pelo menos uma custódia terminal, vai puxar N IDs para Python antes de re-enviar como IN (...) para Postgres. A Neon cobra por I/O. A query correcta é uma subquery.

Usar subquery sem materializar: occ_qs.exclude(id__in=closed_occurrence_ids) (remover o list) — o ORM gera WHERE id NOT IN (SELECT ...). Melhor ainda: NOT EXISTS via ~Exists(...).

FQ-08 healthcheck não tem throttle — e expõe estado da DB anonimamente Médio

core/views.py — healthcheck

É comum e defensável deixar /health/ aberto para o probe do Fly.io. Mas como está (require_safe + sem throttle + devolve 503 quando a DB está em baixo) é um oracle trivial para um atacante saber quando a Neon está em manutenção. Combinado com força bruta em /api/auth/login/ durante uma janela de downtime, é leak de sinal.

Também: JsonResponse não inclui Cache-Control: no-store. Alguns proxies cacheiam 200s por defeito; o probe do Fly faz 4/minuto com o mesmo Host header — não é problema em Fly mas é má higiene.

Aplicar @cache_control(no_store=True). Considerar um endpoint /health/deep/ gated por token partilhado com o monitor se quiserem detalhes.

FQ-09 O exception_handler customizado devolve message_dict bruto — potencialmente expõe estrutura interna Médio

core/exceptions.py

Quando o Django levanta ValidationError com message_dict, o handler devolve-o tal qual. Mas o message_dict de Django inclui chaves que não existem no serializer público — por exemplo, o validador de imagem em validate_image_max_size ou erros de _validate_type_specific_data podem ter chaves __all__ ou nomes de campos internos (parent_evidence numa situação onde o serializer só expôs o ID). É pequeno, mas útil para um atacante mapear o modelo interno.

Filtrar __all__ para uma chave genérica (non_field_errors) antes de devolver.

FQ-10 CorrelationIDMiddleware aceita X-Correlation-ID do cliente sem validar Médio

core/middleware.py — CorrelationIDMiddleware.__call__

Reutilizar o correlation ID do cliente é um padrão legítimo (tracing entre serviços). Mas aqui o valor é gravado directamente no AuditLog (correlation_id = get_correlation_id() em audit.py) e o campo da DB é CharField sem max_length visível no excerto. Um atacante pode enviar um header de 64KB e poluir logs; um atacante mais paciente pode enviar um UUID válido mas correspondente ao correlation ID de outro utilizador, misturando os rastos.

Aceitar o header só se for um UUID válido (uuid.UUID(value)) — caso contrário gerar um novo. Considerar não confiar de todo em headers externos para auditoria forense.

FQ-11 A CSP autoriza cdnjs.cloudflare.com sem integrity Médio

core/middleware.py — ContentSecurityPolicyMiddleware._build_policy frontend/templates/_leaflet_head.html (provável)

Incluir cdnjs em script-src/style-src/img-src/font-src é razoável para Leaflet, mas sem SRI um comprometimento da CDN (ou do package via typosquat) injecta JS no ForensiQ com a bênção da CSP. Não foram vistos os templates de leaflet_head, mas vale verificar.

Adicionar integrity="sha384-..." e crossorigin="anonymous" a todos os scripts e stylesheets servidos via cdnjs. Considerar self-hosting dos 2–3 ficheiros do Leaflet.

Baixo

FQ-12 Occurrence.clean() muta self.number (side-effect num validador) Baixo

core/models.py — Occurrence.clean

Normalizar espaços dentro do clean() mistura validação com canonicalização. Funciona, mas passa a ser invisível no serializer (se alguém migrar save para não chamar full_clean futuro, a normalização some). Melhor fazer isto num validate_number de serializer ou num save explícito.

FQ-13 DigitalDevice.imei aceita string vazia via regex ^(\d{15})?$ mas não passa pelo Luhn Baixo

core/models.py — DigitalDevice.imei

A validação de IMEI em Evidence.type_specific_data usa validate_imei (com Luhn). Em DigitalDevice.imei usa só um RegexValidator de 15 dígitos. Dois caminhos para o mesmo dado, validações diferentes. Um IMEI inválido-por-Luhn entra pela porta de DigitalDevice.

Substituir o RegexValidator por validate_imei, mantendo o blank.

FQ-14 EvidenceVINLookupView devolve VIN normalizado mas não armazena o snapshot Baixo

core/views.py — EvidenceVINLookupView

O endpoint devolve um URL para o agente abrir numa nova aba. Não há log de auditoria (log_access não é chamado) — ao contrário do IMEI (que também não regista, ver abaixo). Para ISO 27037 é útil saber que o agente X consultou VIN Y às Z horas, mesmo que a resposta seja só um URL.

Registar AuditLog com action VIEW e resource_type custom (ou EVIDENCE quando houver uma evidência pai conhecida).

FQ-15 Redirects singular → plural perdem query strings e são 301 permanente Baixo

core/frontend_views.py — *_singular_redirect

HttpResponsePermanentRedirect é cacheado agressivamente por browsers e proxies. Se reverterem a convenção (pouco provável) ficam presos. E os redirects não propagam a query string — um bookmark antigo /occurrence/?status=open perde o filtro.

Usar 302 durante pelo menos um ciclo de release; propagar request.META['QUERY_STRING'].

Notas

FQ-16 Lookup de IMEI não é auditado Nota

Nem o endpoint de IMEI nem o de VIN chamam log_access. As consultas externas revelam informação sensível (mapeamento IMEI → marca/modelo; VIN → veículo). Para um processo ISO 27037 é razoável exigir que cada consulta fique registada mesmo quando serve apenas para UI. Ver FQ-14.

FQ-17 DEBUG=True no .env.example Nota

O template .env.example tem DEBUG=True. Um developer que copie para .env sem pensar corre local com debug — ok. O comentário em baixo lembra "Em produção, DEBUG=False", mas o Fly.io já fixa isso em fly.toml, portanto a única vítima realista é um developer a correr python manage.py runserver sem se aperceber. Mencionar explicitamente "NUNCA copiar isto para um servidor com internet pública".

FQ-18 Volume persistente Fly.io é único (forensiq_media) Nota

fly.toml — [[mounts]]

auto_stop_machines = true + min_machines_running = 0 + um único volume: se a aplicação algum dia escalar para mais de uma máquina, só uma montará o volume — os uploads dos outros agentes ficam inacessíveis. Para uma plataforma forense, mover cedo para S3-compatible storage (R2, Tigris) é menos doloroso antes de ter mil evidências.
